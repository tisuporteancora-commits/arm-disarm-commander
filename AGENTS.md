## Project Rules

- Keep the Lovable branch history linear and safe: do not force push, rebase, amend, or squash commits that may already be published.
- Prefer small, atomic commits. Keep the branch in a runnable state after each commit.
- Do not commit `.env`, `.app-data/`, local logs, build output, or generated runtime folders.
- Preserve the TanStack Start file-route conventions. When adding routes, let `src/routeTree.gen.ts` be regenerated by the framework.
- Keep UI routes thin. Form state and presentation stay in `src/routes`; business rules stay in `src/features`.
- Keep server-only integrations inside `*.server.ts` modules and import them only from server functions or server route handlers.
- Validate request bodies with schemas before calling use cases.
- Auth checks belong at the API/server-function boundary, not only in page navigation.
- Commands sent to the external alarm system must be logged with success and failure details.
- Use explicit types for domain data. Avoid `any` except in generated files.
- Comments must be in English when comments are needed.

## Local Runtime

- Use Node 22 or newer. Current TanStack dependencies require a newer Node than 20.18.
- Start the app with `npm start`.
- Runtime configuration comes from `.env` and server-side admin settings.
- `.app-data/alarm-state.json` is created automatically at runtime for local settings and command logs. It is not a database and must not be committed.

## Validation

- Run `npm run build` before handing off changes.
- Run focused ESLint on touched TS/TSX files when the global lint is noisy because of generated/template files.
